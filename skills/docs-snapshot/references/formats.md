# Snapshotファイルフォーマット

## 共通ルール

- 見出しの前後に空行を入れる
- 見出しに番号を使わない
- アスタリスクを使わない

## 技術snapshot

### features.md

機能単位で現在の状態を書く。1機能は見出し+箇条書き数行。
コンポーネント名、関数名、ルートパスは書かない。

```md
# 機能一覧

## {機能名}
- {何ができるか1}
- {何ができるか2}
- ステータス: {状態の種類}

## {機能名}
- {何ができるか1}
- {何ができるか2}
```

ルール:
- 見出しは機能名(日本語)
- 箇条書きは「何ができるか」を書く。「どう実装されているか」は書かない
- 1機能あたり3-8行
- 廃止された機能は書かない

### user-flows.md

ユーザー種別ごとに主要導線を番号付きステップで書く。
E2Eテストを生成するときにこのファイルから導線を読む。

```md
# ユーザーフロー

## {ユーザー種別}導線

### {導線名}
1. {ステップ}
2. {ステップ}
3. {ステップ} or {別の選択}
4. {ステップ}
```

ルール:
- ユーザー種別ごとにセクションを分ける
- 導線は「ユーザーが何をするか」で書く。「システムが何をするか」は書かない
- 番号付きステップで書く。箇条書きにしない
- 各ステップは1行。長い説明を入れない
- 分岐がある場合は「A or B」で書く
- 画面遷移が分かる粒度で書く(ページ名やURLは不要)

### architecture.md

システム構成を書く。技術選定の理由はindex.mdに書くのでここには書かない。

```md
# Architecture

## Runtime
{ランタイム環境}

## Rendering
{レンダリング方式}

## Data Fetching
{データ取得方式}

## Authentication
{認証方式}

## Styling
{スタイリング方式}
```

ルール:
- セクション: Runtime, Rendering, Data Fetching, Authentication, Styling, Caching, Error Tracking, Analytics等
- 各セクション1-3行。長文を書かない
- システム構成図はmermaid flowchartで書く
- 選定理由は書かない(index.mdの技術スタックに書いてある)

### sitemap.md

全ページをフラットな箇条書きで書く。

```md
# Sitemap

## Public
- / - {説明}
- /about - {説明}

## Auth Required
- /dashboard - {説明}
- /settings - {説明}

## {権限別セクション}
- /{path} - {説明}
```

ルール:
- セクション分けはアクセス権限で行う(Public, Auth Required等)
- 形式: `- /path - 説明`
- 動的パラメータは`:param`表記
- 多言語バリアント(/en/...)は本体と別に書かない。注記で「全公開ページに/en/あり」とだけ書く
- 廃止パス、リダイレクトは末尾に別セクションで書く

### domain-model.md

ドメインが複雑な場合のみ作成する。

```md
# Domain Model

erDiagram(mermaid)で関係図を書く。

## 集約
- {集約ルート}: {子エンティティ}を含む。{ルート}なしに存在できない
```

ルール:
- 関係図はmermaid erDiagram
- 集約のルールを明記する
- 属性は書かない(コードが仕様)
- glossary.mdの用語と一致させる

### api-schema.md

APIがある場合のみ作成する。

ルール:
- 認証方式とエンドポイントの構造を書く
- 個別エンドポイントの一覧は書かない
- GraphQLの場合: スキーマの主要な型とクエリ/ミューテーションの概要

### components.md

UIコンポーネントが多い場合のみ作成する。

ルール:
- コンポーネント名と用途を1行で書く
- props、バリアント、スタイルの詳細は書かない
- カテゴリ(Layout, Form, Navigation等)で分ける

## 製品仕様snapshot

### 目的

お問い合わせ対応や社内の仕様確認でAIが完璧に回答できるレベルの資料。「今どうなっているか」を正確に記述する。

### ファイル構成の決め方

ファイル構成は固定しない。情報源を全て読み込んだ後、製品の特徴と規模に応じて動的に決定する。

判断基準:
- 問い合わせで「この質問にはこのファイルを見れば答えられる」という単位で分ける
- 1ファイル200行以内を目安にする
- 関連するルール・制約が1箇所にまとまるようにする
- 製品の複雑な部分ほど細かくファイルを分ける

例えば、決済と返金のルールが複雑なら専用ファイルにする。シンプルなら他のトピックとまとめてもよい。

### 文体

箇条書きを避け、繋がりのある文章で書く。読み手が流れを追いながら理解できる形にする。

```md
リクエストはクライアントがクリエイターのプランを選択して送信する。この時点では決済は発生しない。クリエイターがリクエストを承諾した時点で初めて決済が実行される[^1]。承諾後の返金は原則として認められていない[^2]。

[^1]: ガイド「リクエストが承諾された時に決済されます」
[^2]: リクエスト機能規約 第6条
```

NG例（箇条書き）:
```md
- リクエスト送信時は決済なし
- 承諾時に決済
- 返金不可
```

### リンク

Obsidian Wikiリンク形式を使用する。snapshot内のファイル同士や glossary.md への参照に使う。

```md
詳細は[[glossary]]のリクエストの項を参照。[[決済と返金]]にも関連する記述がある。
```

### 脚注

情報の根拠を脚注で示す。情報源が複数ある場合はそれぞれ個別の脚注にする。

情報源の種類と表記:
- 利用規約: `利用規約 第N条`
- リクエスト機能規約: `リクエスト機能規約 第N条`
- ガイド: `ガイド「セクション名」`
- NSFWガイド: `NSFWガイド`
- インボイスガイド: `インボイスガイド`
- コード: `コード: ファイルパス` （ドメインロジックの根拠を示す場合）
- 内部メモ: `notes/ファイル名`
- バックログ: `backlogs/ファイル名`

### 実装状態の明記

ガイドや規約に記載があるが実装されていない機能は、必ずその旨を明記する。

```md
有償リクエストは送信から8日以内にクリエイターが承諾しなかった場合、キャンセルが可能になる[^1]。ただし、この機能はシステム上まだ実装されておらず、現在は運営が手動で対応している[^2]。

[^1]: リクエスト機能規約 第7条
[^2]: backlogs/cancel-request-by-requester.md
```

### 古い情報の排除

生成時に全ての情報源を最新の状態で読み込み、矛盾がある場合はコードの実装を正とする。notes/ や backlogs/ の日付が古い場合は、現在のコードと照合して現状を正確に記述する。
